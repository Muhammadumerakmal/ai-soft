## 3. Implementation Order

### 3.1 Critical Path
The implementation must follow this critical path exactly:

1. **Monorepo scaffolding** → 2. **Database schema** → 3. **Auth system** → 4. **Project CRUD** → 5. **Orchestrator** → 6. **Agent executor** → 7. **CEO Agent** → 8. **PM Agent** → 9. **Architect Agent** → 10. **Approval gates** → 11. **Frontend dashboard** → 12. **WebSocket streaming** → 13. **Engineering agents** → 14. **MCP server** → 15. **Testing + Deployment**

### 3.2 Parallelisable Work
Once the orchestrator and agent executor are stable (end of Phase 1), the following can be parallelised:
- Frontend dashboard (Phase 2) with WebSocket streaming
- Engineering agent development (Phase 3) can be split across engineers
- MCP server (Phase 4) can be developed independently

## 10. Common Implementation Concerns

### 10.1 LLM Integration
- **Token limits**: Track and enforce per-agent budgets
- **Response parsing**: Use structured output (JSON mode) for reliable parsing
- **Fallback model**: Configure GPT-4o-mini as fallback when GPT-4o is rate-limited
- **Cost monitoring**: Log tokens per project; set daily budget alerts

### 10.2 File Generation
- **Large files**: Stream writes for files > 1MB
- **Directory structure**: Create directories before writing files
- **Format**: Prettier-format generated code before finalising
- **Validation**: Verify generated files compile/parse before marking complete

### 10.3 Error Recovery
- **Agent failure**: Max 3 retries per agent, then mark project as `failed`
- **Pipeline corruption**: Manual recovery endpoint to restart from specific phase
- **Data inconsistency**: Use DB transactions for all state-changing operations

### 10.4 Development Workflow
- **Feature branches**: `feat/{phase}-{feature-name}`
- **Code review**: All PRs require at least 1 approval
- **Testing**: Tests must pass before merge
- **Documentation**: Update docs alongside code changes
